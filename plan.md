# CPHOS 题目数据库 V1 实施规划

## Summary
- 目标是在 2 周内交付一个可持续维护的题库 V1：完成 LaTeX 主索引题库、原始资料整理规范、导入清洗流程、只读查询 API、xlsx 统计工作簿入库、基础统计和难度评定预留。
- V1 固定范围为：理论题与实验题、试卷主 LaTeX、题目 LaTeX、答案 LaTeX、题图、题目来源、题目聚合统计和原始 xlsx 统计表。
- 技术路线固定为：`GitHub 私有源码仓 + 服务器上的 SQLite 主库 + 服务器资产目录 + FastAPI 只读接口 + JSONL/CSV 导出`。
- 题目定位以 LaTeX 文件索引和卷内 `paper_index` 为准，不再以 PDF 页码作为主来源标记。

## 半周推进总览
- 第 1 周上半：把边界、资料、字段和样例格式定死，先让团队对“录什么、怎么录、录到哪”没有歧义。
- 第 1 周下半：把数据库骨架、bundle 校验、导入链路和工作簿入库打通，形成第一个能跑通的最小闭环。
- 第 2 周上半：把 API、统计聚合、质量检查和首批真实数据录入完成，形成可查、可校验、可复跑的内部版本。
- 第 2 周下半：补齐难度预留、运维与交接文档、服务器部署流程和验收材料，形成可交付版本。

## 第 1 周上半：定标、盘点、冻结格式
### 目标
- 统一项目术语、数据边界和录入规范。
- 完成原始资料盘点，明确哪些材料可直接导入，哪些需要人工清洗。
- 冻结 bundle 结构、字段字典和样例录入格式。

### 必做任务
- 写一页项目定义，明确 V1 做什么、不做什么。
- 统一以下术语：届数、试卷类型、主 LaTeX、题目 LaTeX、答案 LaTeX、卷内 `paper_index`、score workbook、聚合统计、状态标签。
- 盘点现有资料目录：试卷主 LaTeX、题目 LaTeX、答案 LaTeX、图片、历史 PDF、统计 xlsx、其他文本材料。
- 为每类资料建立清单，至少包含：资料编号、年份或届数、试卷类型、文件路径、资料类型、是否可解析、是否缺失、是否需要人工整理。
- 明确哪些统计表要整表入库，哪些只是导出表或过渡文件。
- 冻结 `papers`、`questions`、`score_workbooks`、`question_stats`、`difficulty_scores` 的字段和枚举值。
- 给出 1 套完整样例 bundle，覆盖以下情况：
  - 理论题
  - 实验题
  - 带图片题
  - 带答案 LaTeX
  - 带 xlsx 工作簿
- 写清命名规则：`paper_id`、`question_id`、`workbook_id`、图片文件名、LaTeX 路径、题目顺序编号。

### 本半周交付物
- `docs/project_definition.md`
- `docs/field_dictionary.md`
- `docs/ingestion_guide.md`
- 一份完整可参考的 demo bundle
- 一份资料盘点总表

### 验收标准
- 任意组员只看字段字典和样例 bundle，就能知道一套新卷子应该整理成什么样。
- 团队内部不再出现“题干是存文本还是存文件”“统计表是存聚合值还是整表”这类歧义。
- 能列出当前资料中的风险项，例如：LaTeX 缺失、答案分散、图片命名混乱、xlsx 口径不统一。

## 第 1 周下半：骨架落地、导入打通、最小闭环
### 目标
- 建好服务器 SQLite 的表结构和源码仓的运行骨架。
- 打通“bundle 校验 -> 导入数据库 -> xlsx 工作簿入库 -> 查询”这条最小闭环。
- 让组员可以本地或服务器上复跑整个流程。

### 必做任务
- 初始化 SQLite schema，并补齐索引、唯一约束、外键关系。
- 确保 `papers` 保存届数、类型、主 LaTeX、可选 PDF 和 `question_index_json`。
- 确保 `questions` 保存题目 LaTeX 路径、答案 LaTeX 路径、`paper_index`、检索索引、状态和标签。
- 确保 `score_workbooks` 能把 xlsx 文件作为 BLOB 入库，并保存哈希、sheet 名称、文件大小、来源文件名。
- 补齐 bundle 校验规则：
  - 必填字段检查
  - LaTeX 路径存在性检查
  - 图片存在性检查
  - `paper_index` 唯一性检查
  - `question_id`、`workbook_id` 唯一性检查
- 打通导入脚本：
  - dry-run
  - commit
  - 导入日志写入 `import_runs`
- 打通单独的 xlsx 工作簿入库脚本，便于后补历史统计表。
- 完成一轮 demo 数据初始化、导入和导出验证。
- 配置环境变量方式，让源码仓默认跑 demo 库，服务器上跑生产库。

### 本半周交付物
- 可初始化数据库的脚本
- 可校验 bundle 的脚本
- 可导入 bundle 的脚本
- 可单独导入 xlsx workbook 的脚本
- 更新后的 `.env.example`、部署模板和 `.gitignore`
- 一份最小可运行的 demo 数据库

### 验收标准
- 从空库开始，执行脚本后能得到一份包含试卷、题目、图片和 xlsx 工作簿的数据库。
- 错误的 bundle 会被清晰拦截，而不是带病入库。
- 工作簿入库后能查到元数据，不只是聚合统计量。
- 样例链路可以重复执行，结果稳定。

## 第 2 周上半：API、统计层、首批真实录入
### 目标
- 形成内部可用的检索 API。
- 打通题目级统计聚合，并和原始 xlsx 工作簿建立可追溯关联。
- 完成首批真实卷子的录入、校验和问题清单。

### 必做任务
- 完成 FastAPI 只读接口，至少包括：
  - `/papers`
  - `/questions`
  - `/questions/{question_id}`
  - `/score-workbooks`
  - `/score-workbooks/{workbook_id}`
  - `/search`
- 支持常用筛选：届数、试卷类型、题型、状态、是否有答案、是否有图片、标签、关键词、平均分区间。
- 打通统计聚合脚本：
  - 从 CSV 或后续抽取流程生成 `question_stats`
  - 记录 `stats_source`
  - 记录 `stats_version`
  - 关联 `source_workbook_id`
- 建立基础质量检查脚本，至少检查：
  - 缺失题目 LaTeX
  - 缺失答案 LaTeX
  - 缺失主试卷 LaTeX
  - 缺失图片
  - 缺失 workbook blob
  - 重复题号
- 开始导入首批真实试卷，优先覆盖最典型的理论题和实验题。
- 每导入一套真实卷子，都执行一次质量检查并更新缺失项清单。
- 为录入中的题目补状态标签，区分 `raw`、`reviewed`、`published`。

### 本半周交付物
- 可访问的内部 API
- 首版统计入库流程
- 质量检查报告模板
- 第一批真实试卷导入结果
- 缺失项与异常项清单

### 验收标准
- 组员无需直接打开 SQLite，就能通过 API 查询试卷、题目和工作簿元数据。
- 任意一条统计记录都能追溯到对应的 xlsx workbook。
- 至少完成若干套真实卷子的录入，并知道剩余未录入的原因。
- 质量检查报告能明确指出缺什么、错在哪、下一步怎么补。

## 第 2 周下半：难度预留、部署交付、最终验收
### 目标
- 完成 V1 的最终收口，让项目具备上线到自有服务器并交给组员长期维护的条件。
- 把后续难度评定的扩展位、运维方式和日常操作手册写完整。

### 必做任务
- 接入基础难度脚本，先根据平均分率、零分率、满分率等生成 baseline 难度。
- 在 `difficulty_scores` 中写入算法版本、特征摘要和置信度字段。
- 明确文档说明：未来如果要做更复杂的难度模型，需要哪些更细粒度数据。
- 完成服务器部署说明：
  - 源码仓如何同步到服务器
  - 环境变量如何配置
  - 服务如何启动和重启
  - 日志放在哪里
  - 数据库和工作簿如何备份
  - 出问题如何回滚
- 完成维护手册：
  - 新增一套 LaTeX 试卷怎么整理
  - 新增 xlsx workbook 怎么入库
  - 统计怎么导入
  - API 怎么本地启动和服务器启动
  - 质量检查怎么跑
  - 导出怎么做
- 做一次完整验收演练：从一套新的试卷资料开始，按文档走完整流程。
- 整理最终交付清单，标记哪些是源码仓内容，哪些是服务器运行态内容。

### 本半周交付物
- baseline 难度脚本和样例输出
- 完整的部署文档与维护手册
- 最终质量检查报告
- 最终验收清单
- 可对外说明的 V1 范围和后续扩展说明

### 验收标准
- 未参与开发的组员只看文档，也能独立完成一次新增试卷和统计表入库。
- 服务器上可以稳定运行 API，且源码仓里不暴露真实服务器地址、路径、账号或密钥。
- 能明确回答以下问题：
  - 一道题对应哪个 LaTeX 文件
  - 这道题属于哪套卷子、卷内第几题
  - 这条统计来自哪个 workbook
  - 当前难度值是怎么来的
  - 如果数据错了应该去哪里修

## 每半周固定检查项
- 每半周结束时，更新一次字段字典、录入规范和计划状态。
- 每半周结束时，输出一次“已完成 / 未完成 / 风险 / 下一半周安排”的简短周报。
- 每半周至少跑一次完整回归：bundle 校验、数据库初始化、导入、统计、难度、质量检查、单元测试。
- 每次新增真实数据时，都要保留来源路径、导入时间和导入批次。
- 生产数据库、原始工作簿、导出结果和 `.env` 一律不提交到源码仓。
